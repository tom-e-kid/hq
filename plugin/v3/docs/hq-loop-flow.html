hq plugin v3 — orchestrated workflow

/hq:loop フロー全体図

入力（hq:task または自由テキスト）から、計画 → 実装 PR → 自動トリアージ → 確認済み hq:feedback までを 1 コマンドで駆動する。ユーザが手を動かすのは 3 箇所だけ。それ以外は main conversation と 2 つの subagent が自律実行する。

ユーザ介入点（非スキップ） main conversation で自律実行 subagent で自律実行

介入点: ① Stage 1 の brainstorm + go ゲート / ② executor の consult-needed 往復（Self-Review が significant-gap のときのみ・稀） / ③ Stage 4 の feedback 確認

STAGE 1

Draft — 計画を作る

interactivemain conversation
spec: rules/draft-protocol.md（/hq:draft と同一プロトコルを inline 実行）
  1. P1
    Intake + 影響範囲サーベイ hq:task があれば取得（hq:wip ゲート）。セッション前の会話文脈を引き継ぎ、3 本のサーベイを必ず実行 — 過去 commit（git log）/ 関連マージ済み PR / 中心シンボル grep。ゼロ件も明示報告。
  2. P2
    Brainstorm + Simplicity gatekeeper対話 探索主導の対話。ゲート質問: 再利用 vs 新規 / 最小解比較 / 波及コスト / plan 分割（結合度テスト）。[primary] は最強 tier の start 実行可能シグナルに収束するまで exit しない（hedging 禁止）。
  3. P3
    Compose → consumer coverage check → commit-or-pushback gatego ① plan body を全文 verbatim で提示。go で確定 / 違和感なら P2 に戻って再収束（再提示必須）。承認した本文がそのままファイルになる — drift なし。
  4. P4
    plan ファイル作成 title から branch 名を導出（<type>/<slug> ≤40 字）→ .hq/tasks/<branch-dir>/plan.md + context.md（source / branch）を書く。git branch はまだ作らない。
  5. P5
    Reportファイルパスと branch 名を確定。plan はこの時点で直接編集可能。
branch 名をキーに executor を起動（subagent_type: hq:executor / mode: agent / iteration + budget を prompt で渡す）
STAGE 2

Execute — 実装して PR を作る

executor agentmodel: inherit
spec: rules/start-protocol.md — agent mode（/hq:start と同一プロトコル。逸脱は「ユーザに聞けない」ことへの対処のみ）
  1. P1
    Pre-flightbranch 解決（find-plan.sh）。git branch が既にあれば auto-resume — checkbox 状態から再開フェーズを決定。
  2. P2
    Load planローカル plan.md + context.md を読む。source: があれば hq:task を fetch。
  3. P3
    Execution prepbase 解決 → git checkout -b → context.md に base_branch 追記 → gh/task.json → focus をメモリへ。
  4. P4
    Execute（Plan item ごとに 1 commit） .hq/start-memory.md（過去 run の蒸留知見）を先に読む → item ごとに Before Edit → 実装 → Before Commit → checkbox toggle → commit。詰まったら FB を書いて続行（continue-report）。
  5. P5
    Acceptance sweep（検証のみ・修正しない） [auto] 項目を 1 件ずつ実行・toggle（バッチ禁止）。失敗 → P4 loopback で修正（item ごと retry cap 2）→ 再 sweep。cap 超過は FB + [x]-anyway。[primary] 失敗は最上位で報告。
  6. P6
    Self-Review（3 軸の自己評価）consult ② Plan 整合 / 範囲外影響 / トンネルビジョン。pass → 続行、minor-gap → FB、significant-gap → consult-needed を返して停止 — orchestrator がユーザに質問 → 回答を載せて再起動（auto-resume で P6 に着地）。
  7. P7
    Quality Review（pure review — 自動修正なし） Step 1: judgment mode で agent 選抜（credential 検出は security-scanner 強制）。Step 2: code-reviewer / security-scanner / integrity-checker を並列起動。全 finding は FB 化されるだけ — 修正判断は人間へ。
  8. P8
    PR 作成 ゲート（全 checkbox）→ workflow sections pack を組み立て: ## Manual Verification / ## Known Issues（3 カテゴリ + FB→done/ をアトミック移動）/ ## Implementation Plan（plan 全文スナップショット = 恒久記録）/ Refs #task。pr skill の手順を read-and-follow で実行。
  9. P9
    Retrospective.hq/retro/<branch-dir>.md — FB ごとの検出妥当性 / 実装時予防可能性 + P6/P7 判断の hindsight。
  10. P10
    Distillationretro から repo 固有の教訓を .hq/start-memory.md（1500 字上限）へ蒸留 → 次 iteration の P4/P6/P7 が読む（学習ループ）。
  11. P11
    構造化リターンstatus / pr_url / fb_summary / phase_timing / distilled_learnings — orchestrator がレポートに整形。
PR 番号 + 残り iteration budget を渡して auto-triager を起動（subagent_type: hq:auto-triager）
STAGE 3

Auto-triage — Known Issues を自動処理

auto-triager agentmodel: sonnet
spec: rules/triage-protocol.md — auto mode（/hq:triage と同じ liveness + ordered gate。per-item 確認だけを外し、補償柵を入れる）
  1. P1
    Load PRheadRefName → <branch-dir> → ローカル plan を回収。## Implementation Plan の存在を確認。
  2. P2
    Parse Known IssuesTriage summary + Must Address / Recommended / Optional の 3 カテゴリを item 化。
  3. P3
    Liveness → ordered gate で disposition 導出（確認なしで適用） Gate 0 解決済み→2 ／ Gate 1 偽陽性→2 ／ Gate 2 別オーナー→3 ／ Gate 3 自明+低影響→4、実質的→1。バイアス則: 不明は安全側（validity 不明→2、scope 不明→1）。
    auto mode の逸脱: 3（escalate）は絶対に適用しない — candidate として収集し Issue は作らない ／ 1（plan 追記）は budget 残 かつ Must Address のみ（Medium/Low は 2 に降格、budget 切れの Must Address は candidate 化）。
  4. P4
    Apply4 の修正（regression gate: format/build/test、2 回失敗で revert・open 据え置き）→ 1 の plan.md 追記 → PR body を単一 gh pr edit で変換。
  5. P5
    構造化リターンper-item 監査証跡（gate / SHA / 降格ノート）+ feedback_candidates + plan_gained_items。

ループ判定（orchestrator）

↺Stage 2 に再突入 — plan_gained_items: true かつ 再突入回数 < loop_max_iterations（既定 2）。executor は auto-resume で追記された未チェック Plan item から実行。
↓Stage 4 へ — 追記なし、または budget 消化。cap はハード上限: 積み残しの Must Address は feedback candidate として必ず浮上（silent drop 禁止）。
STAGE 4

Report + Feedback 確認 — 締める

interactivemain conversation
spec: commands/loop.md § Stage 4 — ループ内で唯一 hq:feedback Issue を作れる場所
  1. 1
    作業レポート PR URL / iteration ごとの executor サマリ（primary 結果・Self-Review・agent 選抜・FB 数・phase timing）/ triage 監査証跡（各 item → 適用 disposition・SHA）/ 各 run の蒸留知見 — 学習ループが回っているかを可視化。
  2. 2
    Feedback candidates の確認確認 ③ 蓄積された candidate（title / severity / origin / rationale）を提示 → ユーザが multi-select（「作らない」も有効な回答）。選択分のみ gh issue create --label hq:feedback（body に Refs #PR）→ PR の該当行を escalated: #N に変換。却下分は「dropped by user decision」として報告。
  3. 3
    次の一手PR をレビュー / merge → /hq:archive（done）。やめるなら /hq:archive cancel。

ステージを流れる成果物

.hq/tasks/<branch-dir>/plan.md

hq:plan の単一情報源（gitignore 下）。Stage 1 が作り、Stage 2 が checkbox を進め、Stage 3 の disposition 1 が追記する。

PR body

Manual Verification / Known Issues / Implementation Plan（plan 全文スナップショット = 恒久レコード） / Refs #task。Stage 3 が Known Issues 行を変換。

.hq/tasks/<branch-dir>/feedbacks/

実行中の FB ファイル。PR 作成時に ## Known Issues へ昇格し done/ へアトミック移動。

.hq/retro/<branch-dir>.md → .hq/start-memory.md

P9 が retro を書き、P10 が 1500 字に蒸留。次 run の P4（実装）/ P6（Self-Review）/ P7（agent 選抜）が読む学習ループ。
