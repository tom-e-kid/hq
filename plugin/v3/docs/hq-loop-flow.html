/hq:loop フロー全体図

単体コマンド（draft / start / triage）を廃止し、/hq:loop を唯一のパイプライン入口にする統合モデル。設計原理: 決定論で片付かない意味的判断はすべて root agent（loop を受け持つ session model）が下す。決定論レール（スクリプト・構造ゲート・regression gate）は決定論のまま、subagent は証拠収集と実行に徹する。PR は triage 完了後に「最終対応案」として最後に作られる。

ユーザ介入点（非スキップ） root agent の意味的判断（J1–J7・決定記録必須） subagent 実行 決定論ステップ

ユーザ介入は 3 系統: ① Stage 1 の brainstorm + go/stop ゲート ／ ② root が consult を選んだとき — J3(build 受け入れ)・J5(triage)・J8(発散時の計画見直し提案 / 安全キャンセル) ／ ③ Stage 7 の feedback 確認。root の判断 J1–J8 はすべて .hq/tasks/<dir>/reports/ に決定記録を残す。
進捗の常時可視化: loop 開始時に Stage 0–7 を TaskCreate でタスク化し、stage 境界で更新。J8 再突入は Build (iteration 2) のように追加、長い stage は Build — item 3/5 と件数更新、subagent も同じタスク UI に報告 — タスク一覧を見ればいまループのどこにいるか常にわかる。

STAGE 4

Triage — PR を作る前に裁く

root modelverifier agent（必要時のみ）
  • J5
    FB ごとの disposition 判断ROOT対象はローカル FB ファイル（PR body ではない — PR はまだ無い）。validity → ownership → scope/risk の順序ゲートとバイアス則（不明は安全側）を判断の事前分布として、plan の意図・build の実態・budget・severity に照らして root が決定する。エージェントの機械的手続きではない。証拠不足の FB は読み取り専用の verifier agent を単発起動して補強してよい。
  • ·

USES commands/loop.md §Triage Judgment（順序ゲート + バイアス則の新しい置き場）· rules/feedback.md · .hq/triage.md（override — 判断の事前分布ヒント）

J8収束判定（root）— 毎サイクルの Stage 4 直後に発火（初回含む）。機械的な回数上限ではなく FB の軌跡を意味的に判断する

↓収束 — 残るのは些細な指摘のみ（低 severity・新たな設計論点なし）。初回 triage の FB が全部 micro-fix 相当なら iteration 0 で収束扱い。→ executor の micro-fix パス 1 回（regression gate 必須）→ integrity-checker のみ micro-diff スコープで再走（「些細な修正」が壊しうる唯一の軸 = 残存参照・consumer・surface 整合の再確認）+ root の micro-diff 目視 → Stage 5 へ。Stage 3–4 のフル再走はしない。
↺継続 — 実質的だが有界な対応が残り budget もある → Stage 2 に再突入（Stage 3–4 も再走）。loop_max_iterations（既定 2）は暴走時のハードバックストップとして残る。
⛔発散 — 修正が新たな不具合・検討事項を生んでいる（兆候: 修正済み箇所への同等以上の severity の FB、plan の前提と矛盾する FB、fix→新 FB の連鎖、primary の反復失敗）→ 計画自体の欠陥仮説。ループをブロックし、問題分析 + 計画上の根本原因 + 計画見直し案を verbatim で人間に提示（介入 ②）:
・承認 → plan.md を改訂（影響 item を再オープン）、budget リセット → Stage 2 からやり直し
・却下 / 中止 → 安全キャンセル — archive cancel 経路を read-and-follow（PR は未作成なので task フォルダ → .hq/tasks/canceled/、branch 強制削除、memory クリア）→ 経緯を報告して終了。

成果物と残るコマンド

.hq/tasks/<branch-dir>/plan.md

root と executor の作業台帳（gitignore 下）。PR には埋め込まない — 本質（Why/Approach）は J6 が PR ナラティブへ昇華する。archive で done/ へ。

feedbacks/ → PR ## Known Issues

FB は PR 作成前に J5 で裁かれる。PR に載るのは triage 済み残存（accepted / escalated）のみ。done/ への移動は disposition 確定時。

.hq/tasks/<dir>/reports/

J1–J7 の決定記録。「root が何をどう判断したか」の監査面。Stage 6 の retro と Stage 7 の報告が消費する。

残るコマンド

/hq:loop（唯一のパイプライン入口）・/hq:respond（外部レビューコメント対応）・/hq:archive（done / cancel）。draft / start / triage の単体コマンドと auto-triager agent は廃止。
